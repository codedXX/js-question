### 问题1：为什么proxy里的对象新增属性的时候不需要$set?

当你执行 `userProxy.email = 'test@example.com'` 时，哪怕 `email` 这个属性原本在 `user` 对象中根本不存在，Proxy 的 `set` 陷阱依然会被触发！

在这个 `set` 陷阱里：
- `target` 就是原始的 `user` 对象
- `key` 就是你新增的 `'email'`
- `value` 就是 `'test@example.co'`

#### 总结
- **Vue 2 (`Object.defineProperty`)**：只能劫持属性，不能劫持对象。新增属性需要专门的 API (`$set`) 介入处理。
- **Vue 3 (`Proxy`)**：在对象外层设了一道“关卡”（代理），不论是访问已有属性，还是增加新属性，甚至是删除属性（`deleteProperty` 陷阱），都必须经过这道关卡，因此不再需要 `$set` 这样的补丁 API 了。

### 问题2：为什么叫陷阱呢？这不是很好的设定么？

这是一个非常敏锐的问题！你感觉完全正确，这确实是一个非常强大且优秀的设计。

之所以叫**“陷阱”（Trap），是因为这是一个计算机科学领域的专业术语**，在这里它是中性的，甚至带有几分巧妙，完全没有现实生活中“坑害”、“危险”的贬义色彩。

我们可以从以下几个角度来理解为什么官方规范（ECMAScript）会选用这个词：

#### 1. 计算机底层概念的延伸
在操作系统和底层计算机体系结构中，“Trap”（通常翻译为陷入、陷阱或软件中断）是一个非常常见的概念。它的意思是：当系统执行到某一步，或者发生特定的事件时，强行中断当前的正常执行流程，把控制权“移交”给一段特定的处理程序。

JS 中的 `Proxy` 完美继承了这个概念：当你执行 `proxy.name` 时，JS 引擎原本打算按部就班地去内存里读取原始对象的 `name` 属性。但是，你在中间挖了一个 `get` “陷阱”。JS 引擎一脚踩空，掉进了你事先准备好的处理函数中。

#### 2. 与“拦截”（Intercept）的生动比喻
你可以把 JS 引擎原本对对象的操作（读、写、删除等）想象成在一个平坦大道上正常行驶的汽车。

使用 `Proxy`，就像是你在这条路上设了一个**“关卡”或“暗穴”（Trap）**。
当引擎试图驶过这条路（比如尝试读取属性）时，它会被自动“捕获（catch/trap）”，被迫停下来，执行你在这个关卡里写的自定义代码。
等你的代码执行完了（并且决定放行，比如在 `set` 里 `return true`），它才能继续原来的行程。

#### 3. 如果不叫 Trap，还可以叫什么？
在其他框架或语言中，类似的概念其实有其他更接地气的名字，如果把 Trap 换成它们，你可能会觉得非常亲切：
- **Hook（钩子）**：例如 Vue 的生命周期钩子，或者 React Hooks，意思都是把自定义代码“挂载”到某个固定流程上。
- **Interceptor（拦截器）**：例如 Axios 的请求/响应拦截器，在发送请求前或收到响应后“拦截”并处理一下。
- **Middleware（中间件）**：在到达最终目的地的中间过程进行加工。

#### 总结
在 ECMAScript 规范中，这些被定义在 `Proxy handler` 对象里的拦截方法，其官方标准名称就是 Trap。它代表的是**“捕获默认行为并接管控制权”**的能力。

所以，这不仅不是一个“坑”，反而是赋予开发者上帝视角、让你能够在底层修改语言默认行为的顶级魔法！

### 问题3：Object.defineProperty 和 Proxy 的区别

1. **劫持目标不同**：
   - `Object.defineProperty` 只能劫持对象的某一个确定的属性（代码里写死了 `'name'`）。如果要劫持对象里的所有属性，就必须用 `for...in` 循环遍历对象的所有属性，一个个去调用 `Object.defineProperty`。
   - `Proxy` 劫持的是整个对象（`new Proxy(user, {...})`），所以新增属性也能自动被拦截。

2. **代码复杂度（中转变量）**：
   - `Object.defineProperty` 在写 `get` 和 `set` 时，通常需要一个外部变量（如 `internalName`）来保存真实的值，否则很容易引起栈溢出死循环。
   - `Proxy` 配合 `Reflect`，直接操作对象，代码结构更清晰，不需要借助外部中转变量。

### 问题4：为什么 reactive 解构之后会失去响应性？

这与 JavaScript 本身的**赋值机制**有关，并不是 Vue 的缺陷。

Vue 的 `reactive` 是基于 `Proxy` 实现的。Proxy 的作用是拦截对**代理对象属性**的操作（比如 `state.count = 1`）。

当你对一个 `reactive` 对象进行解构时：
```javascript
const state = reactive({ count: 0, name: 'Vue' })
let { count } = state
```
这段代码在 JavaScript 引擎执行时，完全等价于：
```javascript
let count = state.count // 这里触发了一次 Proxy 的 get，拿到了基本数据类型的值 0
```

**失去响应性的根本原因：**

1. **基本数据类型按值传递**：因为 `count` 是一个数字 `0`（基本数据类型），上面的操作仅仅是把 `0` 这个值**复制**给了本地的 `count` 变量。
2. **脱离了 Proxy 拦截**：在这之后，如果你执行 `count = 10`，你修改的只是本地的这个普通变量 `count`。你并没有通过 `state.count = 10` 去修改，因此**完全没有经过 Proxy 对象**，自然触发不了 `set` 陷阱，Vue 无法追踪变化，视图也就不会更新。

**例外情况：解构出来的是对象**
如果解构出来的是一个引用类型（对象或数组），因为它是按引用（内存地址）传递的，解构出来的变量指向的依然是 Proxy 代理过的深层对象，所以**不会**失去响应性：
```javascript
const state = reactive({ user: { age: 18 } })
const { user } = state 
user.age = 20 // 依然有响应性！因为 user 还是那个 Proxy 对象，会触发它的 set 陷阱。
```

**Vue 的解决方案：toRefs**
为了在解构时不失去响应性，Vue 提供了 `toRefs` (和 `toRef`)。它的原理是把对象的每一个属性都变成一个包含 `.value` 的 `ref` 对象：
```javascript
import { reactive, toRefs } from 'vue'

const state = reactive({ count: 0 })
// toRefs 把 state 变成了类似 { count: Ref(0) } 的结构
const { count } = toRefs(state) 

count.value++ // 是响应式的！因为你操作的是 ref 对象的 .value 属性，依然在 Proxy/Ref 的拦截范围内，并会同步到原 state。
```


问题5：Promise.resolve().then的返回值是什么？


Promise.resolve().then() 的返回值是一个新的 Promise 对象。
1. 核心结论
无论 .then() 中回调函数的代码是什么，.then() 方法永远会返回一个全新的 Promise。这正是 Promise 能够链式调用（Chaining）的基础。

问题7：在vue里，watchEffect是所有数据更新都会触发这个嘛？
不是所有数据更新都会触发，只有在 watchEffect 回调函数中被访问（读取）过的响应式数据发生变化时才会触发。

工作原理
Vue 会在第一次执行回调时自动追踪其中访问的所有响应式依赖，之后这些依赖变化时才重新执行。


const count = ref(0)
const name = ref('Alice')
const age = ref(25)

watchEffect(() => {
  // 只访问了 count 和 name
  console.log(count.value, name.value)
})

count.value++ // ✅ 触发（被追踪）
name.value = 'Bob' // ✅ 触发（被追踪）
age.value = 30 // ❌ 不触发（未被访问，未追踪）
动态依赖追踪
依赖列表是动态的，每次执行后重新计算：


const show = ref(true)
const a = ref(1)
const b = ref(2)

watchEffect(() => {
  if (show.value) {
    console.log(a.value) // show=true 时追踪 a
  } else {
    console.log(b.value) // show=false 时追踪 b
  }
})

// 当 show 变为 false 后，a 的变化不再触发
与 watch 的对比
watchEffect	watch
依赖声明	自动追踪（运行时）	手动指定
初始执行	立即执行一次	默认不立即执行
获取旧值	不能	可以
适用场景	依赖较多、逻辑简单	需要精确控制
总结
watchEffect = "你用了什么，我就监听什么"，而不是监听所有响应式数据。
